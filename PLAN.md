# Smart Alarm: итоговое ТЗ и план реализации

## 1. Общая цель

Нужно реализовать кроссплатформенное desktop-приложение-будильник на C++ и Qt 6.8.3 с использованием Qt Widgets и CMake.

Приложение работает из системного трея. При запуске, если системный трей доступен, главное окно не показывается. При двойном клике по tray icon открывается окно настроек уведомлений. Если системный трей недоступен, приложение показывает ошибку и завершает работу.

Весь пользовательский интерфейс должен быть на английском языке. Комментарии в коде, если они нужны, должны быть на русском языке.

## 2. Основные сущности

### 2.1 Notification

У каждой нотификации есть:

- `id`: UUID, генерируется один раз, пользователю не показывается.
- `enabled`: пользовательский флаг включения/выключения.
- `message`: текст уведомления.
- `color`: выбранный hex-цвет.
- `volume`: громкость `0..100`, задаётся ползунком.
- `playCount`: количество повторов звукового паттерна, `0..999`.
- `sound`: preset или custom pattern.
- `schedule`: одно из расписаний: `once`, `weekly`, `nth_week`, `interval`.

`playCount = 0` означает проигрывать паттерн бесконечно, пока popup не будет закрыт или отложен через snooze.

### 2.2 Global settings

Глобальные настройки:

- `defaultSnoozeMinutes`: целое число `1..1440`, по умолчанию `1`.
- `notificationPosition`: одна из позиций:
  - `top_left`
  - `top_right`
  - `bottom_left`
  - `bottom_right`
  - `center`

Позиция уведомлений является только глобальной настройкой. У отдельных нотификаций переопределения позиции нет.

### 2.3 Runtime-состояния

Некоторые состояния существуют только во время текущего запуска приложения и не сохраняются в JSON:

- глобальный runtime toggle включения/выключения будущих уведомлений;
- `lastTriggeredMinute` для защиты от повторного показа в одной минуте;
- `lastDismissedAt` для `Interval` с `Count from = From confirmation`;
- активные popup-окна;
- pending snooze;
- очередь проигрывания звука.

После перезапуска приложения runtime toggle всегда включён.

## 3. Хранение данных

Все настройки и нотификации хранятся в одном JSON-файле.

Файл должен храниться в стандартном месте настроек приложения для текущей ОС через `QStandardPaths::AppConfigLocation`.

Application metadata:

- organization name: `SmartAlarm`
- application name: `SmartAlarm`
- application display name: `Smart Alarm`

Корневая структура JSON:

```json
{
  "_ver": "1",
  "settings": {
    "defaultSnoozeMinutes": 1,
    "notificationPosition": "top_right"
  },
  "notifications": []
}
```

Поле `_ver` добавляется на будущее, но в текущей версии никак не проверяется и не влияет на логику.

JSON сохраняется сразу после каждого изменения настроек или нотификаций. На выходе из приложения отдельное сохранение не требуется.

Для атомарной записи использовать `QSaveFile`.

### 3.1 Поведение при ошибках JSON

Если файла нет:

- приложение стартует с дефолтным состоянием;
- файл будет создан при первом изменении.

Если весь JSON повреждён:

- приложение не падает;
- повреждённый файл переименовывается в backup с timestamp;
- приложение стартует с дефолтным состоянием.

Если валидный JSON содержит невалидные notification-записи:

- невалидные записи отбрасываются;
- приложение сразу сохраняет очищенный JSON.

Если глобальные настройки содержат невалидные значения:

- невалидные поля заменяются дефолтами;
- валидные нотификации сохраняются;
- исправленный JSON сразу записывается.

Для диагностических сообщений использовать `qWarning()` / `qInfo()`. Отдельный лог-файл в первой версии не нужен.

## 4. Форматы JSON

### 4.1 Notification

```json
{
  "id": "uuid",
  "enabled": true,
  "message": "Stand up",
  "color": "#D94841",
  "volume": 70,
  "playCount": 0,
  "sound": {
    "type": "preset",
    "preset": "gentle_chime"
  },
  "schedule": {
    "type": "once",
    "date": "2026-05-17",
    "time": "10:55"
  }
}
```

Optional-поля не записывать пустыми строками. Если значение не задано, поле лучше опустить.

### 4.2 Sound

Preset:

```json
{
  "type": "preset",
  "preset": "gentle_chime"
}
```

Custom:

```json
{
  "type": "custom",
  "pattern": "880/150, _/50, 660/150"
}
```

Если пользователь переключился с `Custom pattern...` на обычный preset, custom pattern не нужно сохранять.

### 4.3 Once schedule

```json
{
  "type": "once",
  "date": "2026-05-17",
  "time": "10:55"
}
```

### 4.4 Weekly schedule

```json
{
  "type": "weekly",
  "days": ["mon", "wed"],
  "time": "10:30",
  "dateRange": {
    "start": "2026-05-01",
    "end": "2026-06-01"
  }
}
```

Если `dateRange` не используется, его не записывать.

### 4.5 Nth Week schedule

```json
{
  "type": "nth_week",
  "everyWeeks": 2,
  "weekday": "mon",
  "time": "10:30",
  "referenceDate": "2026-05-17",
  "endDate": "2026-12-31"
}
```

Если `endDate` не задан, поле не записывать.

### 4.6 Interval schedule

```json
{
  "type": "interval",
  "everyMinutes": 40,
  "from": "10:00",
  "to": "18:00",
  "countFrom": "trigger",
  "snoozeMinutes": 0,
  "schedule": {
    "days": ["mon", "tue"],
    "dateRange": {
      "start": "2026-05-01",
      "end": "2026-06-01"
    }
  }
}
```

`countFrom`:

- `trigger`
- `confirmation`

`snoozeMinutes = 0` означает использовать глобальный `defaultSnoozeMinutes`.

Если optional `schedule` не используется, его не записывать.

## 5. Главное окно настроек уведомлений

Окно открывается двойным кликом по tray icon или пунктом tray menu.

При закрытии крестиком окно скрывается обратно в tray. Приложение не завершается.

Если окно скрыто, двойной клик по tray icon показывает его. Если оно свернуто, восстанавливает. После этого вызываются `raise()` и `activateWindow()`.

Рекомендуемые размеры:

- стартовый размер: `900x520`;
- минимальный размер: `720x420`.

### 5.1 Верхняя панель

Слева сверху:

- кнопка `New notification`.

Справа сверху:

- кнопка глобального runtime toggle уведомлений, иконка будильника;
- кнопка дополнительных настроек с иконкой шестерёнки.

Глобальный toggle:

- управляет только будущими срабатываниями;
- не закрывает уже открытые popup-окна;
- не останавливает уже играющий звук;
- не сохраняется в JSON;
- после перезапуска всегда включён;
- синхронизирован с tray menu checkbox.

### 5.2 Таблица уведомлений

Использовать `QTableView` + собственную `QAbstractTableModel` и делегаты.

Колонки:

- enabled checkbox, без текстового заголовка;
- color swatch, без текстового заголовка;
- `Message`;
- `Schedule`;
- `Actions`.

В колонке `Actions` две icon-only кнопки:

- edit;
- delete.

Поведение:

- переключение `enabled` сразу сохраняет JSON;
- создание/редактирование сохраняет JSON после успешного `Save`;
- удаление показывает стандартный `QMessageBox` подтверждения и после подтверждения сразу сохраняет JSON;
- `Message` в таблице однострочный, с ellipsis; полный текст показывать в tooltip.

Для просроченного `Once`:

- запись остаётся в списке;
- `enabled` остаётся обычным пользовательским флагом;
- `Schedule` отображается красным цветом и зачёркнутым шрифтом;
- отдельный `completed`-флаг не хранится;
- состояние просроченности вычисляется по дате и времени.

## 6. Диалог редактирования нотификации

Редактор нотификации - модальный `QDialog`.

Изменения применяются только по `Save/OK`. `Cancel` или закрытие окна не меняют запись.

Рекомендуемые размеры:

- стартовый размер: `640x620`;
- минимальный размер: `560x520`.

### 6.1 Общая часть

Сверху:

- поле `Message`;
- справа от него checkbox `Enabled`.

Ниже:

- вкладки расписаний:
  - `Once`
  - `Weekly`
  - `Nth Week`
  - `Interval`

Ниже контента вкладки:

- `Color`: палитра примерно из 20 предустановленных цветов;
- выбранный цвет обведён чёрной рамкой с padding;
- `Sound`: select box с preset-ами и справа icon-only кнопка `Play`;
- `Volume`: классический slider;
- `Play count (0 = until dismissed)`: числовое поле.

Индивидуальное поле `Snooze` показывается только во вкладке `Interval`.

### 6.2 Default-значения новой нотификации

- `enabled`: `true`
- `message`: пустая строка
- тип расписания: `Once`
- `Once date`: сегодняшняя дата
- `Once time`: пустое поле
- цвет: первый цвет палитры, `#D94841`
- sound preset: один из мягких default preset-ов, например `gentle_chime`
- `volume`: `70`
- `playCount`: `0`
- `Interval every`: `40`
- `Interval from/to`: `00:00` / `23:59`
- `Count from`: `From trigger`
- `Interval Snooze`: `0`

### 6.3 Валидация при сохранении

Если обязательные поля не заполнены или невалидны:

- запись не сохраняется;
- диалог остаётся открытым;
- проблемные поля подсвечиваются красным.

Общие правила:

- `Message` не должен быть пустым.
- `Play count`: `0..999`.
- `Volume` не требует отдельной пользовательской валидации, так как задаётся slider-ом.
- `Color` не требует отдельной пользовательской валидации, так как всегда выбран.
- Если выбран `Custom pattern...`, custom pattern обязателен и должен полностью валидироваться.

## 7. Вкладки расписаний

Все даты и времена отображаются независимо от системной локали:

- date: `yyyy-MM-dd`
- time: `HH:mm`
- 24-часовой формат.

Расписания работают в локальном времени системы. В JSON даты и времена хранятся как локальные значения без timezone.

### 7.1 Once

Поля:

- date;
- time.

Дата выбирается только через календарь. Ручной ввод даты запрещён.

Time вводится через специальное поле времени.

Обязательные поля:

- date;
- valid time.

Если дата-время в прошлом:

- запись сохраняется как обычная;
- планировщик не ставит её в очередь;
- в таблице `Schedule` отображается красным и зачёркнутым.

После срабатывания `Once` запись не удаляется, не выключается и не получает `completed`-флаг. Просроченность всегда вычисляется программно.

### 7.2 Weekly

Поля:

- days: кнопки `Mon`, `Tue`, `Wed`, `Thu`, `Fri`, `Sat`, `Sun`;
- time;
- collapsible блок `Date range (optional)`:
  - `Start date`;
  - `End date`.

Выбранные day-кнопки закрашены, невыбранные не закрашены.

Можно выбрать несколько дней.

Обязательные поля:

- хотя бы один день недели;
- valid time.

Date range:

- `Start date` и `End date` включительные;
- если обе даты пустые, range не используется;
- если заполнена хотя бы одна дата, используется только соответствующая нижняя/верхняя граница;
- если `Start date > End date`, ошибку не показывать и сохранять разрешено, но расписание фактически не будет срабатывать.

### 7.3 Nth Week

Поля:

- `Every`: numeric input `1..999`;
- текст `weeks on`;
- weekday select box;
- `Time`;
- `Reference date`;
- `End date (optional)`.

`Time` и `Reference date` находятся на одной строке.

Семантика:

- `Reference date` задаёт начало сетки недель.
- Неделя считается календарной неделей с понедельника по воскресенье.
- Срабатывание происходит в выбранный weekday в тех неделях, номер которых кратен `N` относительно недели `Reference date`.
- `End date` включительная.
- Если `N = 1`, это еженедельное срабатывание в выбранный день, начиная с недели `Reference date`.
- Если выбранный weekday в reference-неделе раньше самой `Reference date`, первое срабатывание в этой неделе не происходит.
- Если `End date < Reference date`, сохранять можно, но срабатываний не будет.

Обязательные поля:

- `Every >= 1`;
- weekday;
- valid time;
- reference date.

### 7.4 Interval

Поля:

- `Every`: numeric input `1..1440`;
- текст `minutes`;
- `From`;
- `To`;
- radio group `Count from`:
  - `From trigger`;
  - `From confirmation`;
- `Snooze (minutes, 0 = use global default)`: numeric input `0..1440`;
- collapsible блок `Schedule (optional)`:
  - days;
  - `Start date`;
  - `End date`.

`From` и `To` включительные.

Если `From > To`:

- ошибок не показывать;
- сохранить можно;
- во время runtime-проверки такое расписание просто не проходит проверку интервала и не показывает уведомление.

Ночные интервалы через полночь не поддерживаются.

`Count from = From trigger`:

- сетка строится от `From` внутри каждого подходящего дня.
- пример: `Every 40 min`, `From 10:00`, `To 13:00` даёт `10:00`, `10:40`, `11:20`, `12:00`, `12:40`.

`Count from = From confirmation`:

- до первого dismiss в текущей сессии используется дневная сетка от `From`;
- после dismiss следующий запуск считается от момента закрытия popup слайдером;
- snooze не считается confirmation;
- `lastDismissedAt` хранится только в памяти и сбрасывается при перезапуске.

Если следующий `From confirmation`-запуск выходит за пределы `From..To`:

- не показывать его;
- ждать следующего подходящего дня/окна.

Optional `Schedule`:

- collapsible состояние является только UI-декором и не сохраняется как смысловой флаг;
- если дни не выбраны и даты пустые, schedule не используется;
- если выбран хотя бы один день или задана хотя бы одна дата, schedule используется;
- если schedule используется, должен быть выбран хотя бы один день, иначе ошибка при сохранении;
- если нужно ограничить только диапазоном дат, пользователь всё равно должен явно выбрать все нужные дни;
- `Start date` и `End date` включительные;
- если `Start date > End date`, сохранить можно, но срабатываний не будет.

## 8. Date/time поля

### 8.1 Date fields

Даты выбираются только через календарь. Ручной ввод даты запрещён.

У всех date fields есть кнопка `X` для очистки значения, включая обязательные поля.

Если обязательная дата очищена, сохранить запись нельзя, поле подсвечивается красным.

Если пустое date field открывает календарь, стартовая дата календаря - текущая системная дата.

### 8.2 Time fields

У всех time fields есть кнопка `X` для очистки значения.

Пустое поле времени при потере фокуса остаётся пустым.

Правила ввода:

- вводятся цифры и первый `:`;
- остальные нецифровые символы игнорируются, кроме случая автоподстановки `:`;
- при любом вводе или вставке берутся только первые 4 значимые цифры;
- если введено 4 цифры, на blur между часами и минутами ставится двоеточие;
- если введено 5+ цифр, учитываются только первые 4;
- при вводе 2 цифр и нажатии любой нецифровой клавиши подставляется `:`.

Нормализация на blur:

- часы clamp в `00..23`;
- минуты clamp в `00..59`;
- результат всегда `HH:mm`;
- `2` -> `02:00`;
- `11` -> `11:00`;
- `299` -> `23:09`;
- `1130` -> `11:30`;
- `2999` -> `23:59`;
- `2460` -> `23:59`;
- `0061` -> `00:59`;
- `2:7` -> `02:07`;
- `2:` -> `02:00`;
- `:7` -> `00:07`;
- `:` -> `00:00`.

## 9. Дополнительные настройки

Окно дополнительных настроек - модальный `QDialog`.

Сохраняет изменения только по `OK`. `Cancel` или закрытие крестиком ничего не меняют.

Рекомендуемые размеры:

- стартовый размер: `420x320`;
- минимальный размер: `360x280`.

Поля:

- `Default snooze interval (minutes)`: numeric input `1..1440`, default `1`;
- `Notification position`: схема экрана с пятью кликабельными квадратиками:
  - top-left;
  - top-right;
  - bottom-left;
  - bottom-right;
  - center.

Default position: `top_right`.

## 10. Tray

Tray menu:

- `Notification settings`;
- checkbox `Enabled` для глобального runtime toggle;
- `Exit`.

Tray icon states:

- обычная иконка, когда runtime toggle включён;
- та же иконка с красной диагональной линией, когда runtime toggle выключен.

Индивидуальные выключенные нотификации на tray icon не влияют.

`Exit`:

- полностью завершает приложение;
- закрывает все окна настроек и popup-уведомления;
- останавливает звук;
- не выполняет специальное сохранение JSON, потому что изменения сохраняются сразу при редактировании.

## 11. Single-instance

Нужно защититься от запуска второй копии приложения.

Использовать `QLockFile` в `QStandardPaths::TempLocation`, например `smart-alarm.lock`.

Если вторая копия запущена:

- показать сообщение `Application is already running`;
- завершить вторую копию.

Передавать первой копии команду показать окно в MVP не требуется.

## 12. Планировщик

Используется простой минутный scheduler.

Поведение:

- при старте сразу выполнить проверку текущей минуты;
- затем поставить single-shot timer до следующей границы минуты, где секунды равны `00`;
- дальше повторять проверку в начале каждой минуты;
- после каждого тика пересинхронизироваться на следующую границу минуты, чтобы timer не уплывал.

При каждой проверке:

- пройти по всем валидным нотификациям;
- учитывать глобальный runtime toggle;
- учитывать `enabled`;
- учитывать расписание;
- учитывать pending snooze;
- не показывать одно и то же срабатывание повторно в рамках одной минуты.

Для защиты от дублей хранить в памяти `lastTriggeredMinute` по `notificationId`, где значение - локальная минута `yyyy-MM-dd HH:mm`.

После перезапуска `lastTriggeredMinute` теряется. Пропущенные срабатывания не догоняются.

## 13. Пропущенные срабатывания

Если приложение было выключено:

- `Once`, время которого уже прошло, автоматически не показывается;
- `Weekly`, `Nth Week`, `Interval` не догоняют пропущенные срабатывания;
- планируется только следующее будущее срабатывание;
- pending snooze после перезапуска не восстанавливается, так как snooze является runtime-состоянием.

## 14. Snooze

Snooze работает только для конкретного экземпляра уведомления, где была нажата кнопка `Snooze`.

Нажатие `Snooze`:

- закрывает popup этой нотификации;
- останавливает её звук;
- задаёт runtime `snoozedUntil = now + resolvedSnoozeMinutes`;
- если звук этой нотификации был в аудио-очереди, очередь переходит к следующей нотификации.

Для `Once`, `Weekly`, `Nth Week` используется глобальный `defaultSnoozeMinutes`.

Для `Interval`:

- используется индивидуальное `snoozeMinutes`;
- если `snoozeMinutes = 0`, используется глобальный default.

Если resolved snooze почему-то меньше 1 из-за повреждённых данных, использовать fallback `1` минуту.

Snooze-срабатывание:

- должно показаться даже если базовое расписание `Once` уже просрочено;
- должно показаться даже если для `Interval` оно выходит за пределы `From..To`;
- блокируется, если глобальный runtime toggle выключен;
- блокируется, если сама нотификация уже `enabled=false`.

Если нотификация находится в snooze, но наступает новое плановое срабатывание той же нотификации:

- новое плановое срабатывание побеждает;
- старый snooze считается завершённым и отменяется.

Если окно этой нотификации уже открыто:

- новое окно для того же `notificationId` не создаётся.

Если пользователь выключает нотификацию:

- уже открытое popup-окно этой нотификации закрывается;
- звук этой нотификации останавливается и удаляется из аудио-очереди;
- pending snooze этой нотификации отменяется;
- будущие плановые срабатывания блокируются.

Если пользователь редактирует нотификацию:

- уже открытое popup-окно остаётся как есть;
- pending snooze отменяется.

Если пользователь удаляет нотификацию:

- открытый popup этой нотификации закрывается;
- звук этой нотификации останавливается;
- pending snooze удаляется;
- запись удаляется из модели и JSON сохраняется.

## 15. Popup-уведомления

Использовать только кастомные popup-окна. Native/system notifications не использовать.

Реализация:

- `QWidget`;
- `Qt::Tool`;
- `Qt::FramelessWindowHint`;
- `Qt::WindowStaysOnTopHint`.

На Linux/Wayland поведение `WindowStaysOnTopHint` может быть ограничено, но для MVP используется стандартное Qt-приближение.

Popup:

- белое окно без шапки;
- внутри жирная мигающая рамка цветом нотификации;
- текст уведомления;
- кастомный `slide to dismiss`;
- небольшая кнопка `Snooze`.

Рекомендуемый размер:

- фиксированная ширина около `360px`;
- высота стабильная;
- под message выделяется фиксированная область;
- текст переносится по словам;
- если текст превышает область, он обрезается без scroll.

Рамка:

- мигает бесконечно, пока окно открыто;
- примерная частота: 500 мс;
- переключается между насыщенным цветом и более светлым/полупрозрачным вариантом.

### 15.1 Slide to dismiss

Сделать кастомный виджет.

Поведение:

- пользователь тянет handle слева направо;
- dismiss срабатывает, если handle отпущен на 90% ширины или дальше;
- если отпущен раньше, handle анимированно возвращается влево.

Dismiss означает подтверждённое закрытие popup.

Для `Interval` с `Count from = From confirmation` именно момент dismiss является confirmation time.

### 15.2 Позиционирование popup

Popup всегда показываются на primary screen.

Использовать `primaryScreen()->availableGeometry()`, а не полный `geometry()`, чтобы учитывать taskbar/menu bar/dock.

Отступы:

- `16px` от края available geometry;
- `12px` между popup-окнами.

Позиции:

- `top_right` / `top_left`: первое окно у верхнего края, следующие ниже;
- `bottom_right` / `bottom_left`: первое окно у нижнего края, следующие выше;
- `center`: стопка вертикально вокруг центра, новые ниже предыдущих с пересчётом всей группы.

Если несколько уведомлений срабатывают одновременно:

- визуально показывать несколько popup-окон стопкой;
- для одного `notificationId` одновременно показывать максимум одно окно.

## 16. Звук

Звук генерируется на лету из pattern strings.

Использовать Qt Multimedia и `QAudioSink`.

Аудиофайлы не нужны.

Volume применяется к сгенерированным PCM-сэмплам или на уровне audio output.

При нескольких активных popup-уведомлениях:

- визуальные окна показываются одновременно;
- звуки не накладываются;
- звуки проигрываются последовательно через общую очередь.

Если у текущей нотификации `playCount = 0`, она занимает аудио-очередь до dismiss или snooze. Следующие звуки ждут.

Если `playCount > 0`, паттерн проигрывается указанное число раз подряд, затем аудио-очередь переходит к следующей нотификации, даже если popup ещё открыт.

### 16.1 Sound presets

Встроенные preset-ы:

- `Classic beep`
- `Double beep`
- `Digital alert`
- `Gentle chime`
- `Urgent`
- `Soft pulse`
- `Custom pattern...`

`Custom pattern...` всегда последний пункт.

Все preset-ы хранятся в коде как pattern strings того же формата, что и custom.

### 16.2 Preview Play

Кнопка `Play` в редакторе:

- проигрывает выбранный pattern один раз с текущей громкостью;
- если preview уже играет, повторное нажатие останавливает текущий preview и запускает заново;
- для custom pattern при невалидной строке подсвечивает поле красным и не играет.

Если в этот момент играет реальный звук уведомления, preview `Play` отключён.

### 16.3 Custom sound pattern

Формат:

```text
frequency/duration
frequency/duration/waveform
_/duration
```

Каждая нота отделяется запятой.

Pause:

```text
_/duration
```

Frequency:

- число в Hz, например `440`;
- note name, например `C4`, `D#5`, `Bb3`.

Duration:

- milliseconds;
- должно быть больше 0.

Waveform:

- `sine` default;
- `square`;
- `triangle`;
- `sawtooth`.

Примеры:

```text
880/150, _/50, 660/150
C5/200, E5/200, G5/300
800/100/square, _/50, 800/100/square, _/50, 600/200/square
E5/300/sine, _/100, B4/300/sine, _/100, G4/400/sine
```

Note reference:

```text
C4=262, D4=294, E4=330, F4=349, G4=392, A4=440, B4=494,
C5=523, D5=587, E5=659, F5=698, G5=784, A5=880, B5=988.
Use # for sharp (C#4=277), b for flat (Bb4=466).
```

Если выбран `Custom pattern...`, рядом с полем должна быть кнопка `?`. При клике открывается окно со справкой по формату.

Custom pattern полностью валидируется при сохранении.

## 17. Палитра цветов

Палитра фиксируется в коде, примерно 20 цветов.

Первый цвет и default color:

- `#D94841`

Палитра должна быть приятной и разнообразной:

- красные;
- оранжевые;
- жёлтые;
- зелёные;
- бирюзовые;
- синие;
- фиолетовые;
- розовые;
- нейтральные.

В JSON хранится выбранный hex.

Если hex из JSON не входит в текущую предустановленную палитру:

- всё равно использовать его в таблице и popup;
- в редакторе добавить его как временный выбранный swatch в конец палитры.

## 18. Формат отображения Schedule в таблице

UI на английском языке.

Примеры строк:

- `Once 2026-05-17 10:55`
- `Weekly Mon, Wed 10:30`
- `Weekly Mon, Wed 10:30, 2026-05-01..2026-06-01`
- `Every 2 weeks on Mon 10:30`
- `Every 40 min, 10:00..18:00`
- `Every 40 min, 10:00..18:00, Mon-Fri`

Для просроченного `Once` эта строка красная и зачёркнутая.

## 19. Автозапуск

Автозапуск при входе пользователя в систему относится к будущему установщику.

В текущей версии приложения автозапуск не реализуется и не настраивается в UI.

## 20. Нефункциональные требования

- Код должен быть чистым, поддерживаемым и эффективным.
- Следовать SOLID и KISS.
- Использовать современные подходы Qt/C++.
- Не смешивать бизнес-логику расписаний с виджетами.
- Удалять неиспользуемый код.
- UI должен быть стабильным, без неожиданных ошибок при некорректных runtime-состояниях или повреждённых настройках.
- Приложение должно работать на Windows, macOS и Linux с учётом разумных ограничений платформ.
